# 01-01-PLAN: Storage Bucket Configuration

## Objective
Establish the `product-images` storage bucket in Supabase with production-grade security and validation (size/type).

## Must Haves
- [ ] `product-images` bucket exists and is set to **Public**.
- [ ] RLS policies allow authenticated admins to `INSERT`, `UPDATE`, and `DELETE`.
- [ ] Bucket-level limits (5MB, images only) are enforced.
- [ ] Verification script `scripts/verify-storage.js` exists to confirm access.

## Tasks

<task wave="1" id="01-01-01">
<action>
Create a verification script `scripts/verify-storage.js`. 
The script should use the `SUPABASE_URL` and `SUPABASE_KEY` from `backend/.env` to check if the `product-images` bucket exists and verify its public access state.
Include a check for the bucket's file size limit (5MB) if possible via the SDK.
</action>
<read_first>
- `backend/.env` (for Supabase credentials)
- `.planning/REQUIREMENTS.md` (for INF-01, INF-02, INF-03)
</read_first>
<acceptance_criteria>
- File `scripts/verify-storage.js` exists.
- The script uses `dotenv` to load environment variables.
- Running `node scripts/verify-storage.js` correctly reports on bucket status.
</acceptance_criteria>
</task>

<task wave="1" id="01-01-02">
<action>
Manual Action: Launch the Supabase Dashboard and configure the Storage bucket.
- Create a new bucket named `product-images`.
- Set the bucket to **Public Access**.
- Under Policies, create RLS policies for `storage.objects`:
  - `Select`: Allow for everyone (Public).
  - `Insert`: Allow for `authenticated` users where `auth.jwt() ->> 'role' = 'admin'`.
  - `Update/Delete`: Same as Insert.
- Set **File Size Limit** to 5MB (5242880 bytes).
- Set **Allowed MIME Types** to `image/*`.
</action>
<read_first>
- `.planning/research/PITFALLS.md` (for RLS policy warnings)
- `.planning/research/ARCHITECTURE.md` (for bucket naming)
</read_first>
<acceptance_criteria>
- Manual confirmation that policies are active.
- `node scripts/verify-storage.js` returns green for bucket existence.
</acceptance_criteria>
</task>

## Verification Criteria
- `node scripts/verify-storage.js` passes all checks for bucket public status.
- Manual test: Accessing any (empty) path on the new bucket returns 200 (if empty) or 404 (if missing), but NOT 401/403 for read access.
